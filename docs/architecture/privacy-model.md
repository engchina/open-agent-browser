# Privacy Model

## Defaults

- The browser agent is local-first.
- Telemetry is off by default.
- Page context is captured only after a user task.
- The side panel applies the selected context policy before publishing or sending a snapshot.
- Memory writes require confirmation.
- Provider mode defaults to disabled.
- Ordinary web page origins cannot call the local API or event stream by default.

## Data Classes

| Data | Storage | Default Handling |
| --- | --- | --- |
| Page snapshot | Extension memory, request body, server process memory, and local SQLite latest-snapshot store | Sanitized and truncated before server/provider use; latest records can be cleared by tab or globally |
| Browser actions | SQLite audit log | Stored locally for task history and safety review; page-changing approvals may store the expected URL used for stale-page checks |
| Tab metadata | Extension memory and local task artifacts | Bounded to tab ID, window ID, index, title, URL, and active state; exposed only after a user task asks for tab context |
| Tab state changes | Extension memory and SQLite audit metadata | Opening or closing tabs requires approval; audit records the reviewed tool metadata without page contents |
| Downloads | Browser download history and SQLite audit/task metadata | Direct URL downloads require approval; task history stores reviewed tool metadata and returned download metadata, not downloaded file bytes |
| Screenshots | Extension memory and redacted local task metadata | Raw image data is available only to the executing extension call; task history stores metadata such as MIME type and byte length, not the base64 image |
| Task runs | SQLite | Stores plan steps and terminal browser tool results, including extension-side error messages |
| Task artifacts | SQLite | Stores reusable local outputs such as extracted links, extracted text, sanitized snapshots, screenshot metadata, and agent output; artifact audit events store metadata only |
| Realtime events | WebSocket clients only | Broadcast locally as task/audit metadata; not sent to providers |
| User memory | SQLite | Written only after confirmation and deletable by memory ID |
| Provider credentials | Environment or browser storage | Redacted from health responses |
| Provider settings | SQLite | API keys are redacted from read APIs |
| New tab launch draft | Extension local storage | Short-lived task text consumed once by the side panel; not sent to the server until the user submits the task |
| Dev extension runtime config | Built extension output | Contains only the loopback agent base URL generated by `pnpm dev:browser`; no credentials or task data |

## Context Policies

- Visible text: sends only sanitized page text, title, URL, and timestamp. Links, element metadata, heading outlines, and table summaries are removed before server publish.
- Interactive elements: sends sanitized text, links, and bounded interactive element metadata. This is the default task mode.
- Full snapshot: sends the largest sanitized snapshot the MVP allows, still with sensitive attributes redacted and size limits enforced.

All policies use the same shared sanitizer. The extension applies the policy before `POST /v1/page/snapshot` and `POST /v1/chat`; the server sanitizes snapshots again on receipt.

For form understanding, the sanitizer preserves non-secret hints such as field names, labels, placeholders, autocomplete values, roles, and input types. Captured input values, secret-like attributes, card-like values, bearer/API keys, and concrete email-address values in attributes are redacted before provider or task-history use.

Snapshot text, element labels, headings, link text, table cells, and page titles are also passed through the same sensitive text redactor before storage or provider prompts. Page URLs and captured link URLs drop embedded credentials and redact sensitive query parameter values such as tokens, API keys, card-like values, and email addresses.

Hidden, inert, and aria-hidden controls are not exposed as actionable snapshot elements. The extension also rejects click/type execution for hidden, disabled, or non-editable selector targets, so provider-supplied selectors cannot bypass the visible-control boundary.

## Provider Modes

- Disabled: no model call is made; `/v1/chat` returns a local snapshot summary and guarded plan.
- Ollama: local HTTP endpoint, default `http://127.0.0.1:11434`.
- OpenAI-compatible: user-provided base URL and API key.

Ollama mode sends no API key or authorization header. It is the default local-provider path for users who want page understanding without cloud API calls.

No provider is contacted unless configured by the user.

The options page can run a provider test. That sends a short connectivity prompt to the submitted provider config and returns a redacted result; the test endpoint does not save the submitted config.

## Local Persistence

The agent server stores confirmed memory, task runs, tool results, task artifacts, audit events, provider settings, and the latest published page snapshots in SQLite. The default database path is `data/open-agent-browser.sqlite`; set `OAB_SQLITE_PATH` to use another location. Pending memory writes and approval tokens remain process-local so a restart clears unconfirmed sensitive operations.

Approval tokens are also bounded by a short local expiry window. Expired tokens are removed from the pending approval registry and cannot authorize or reject a browser action.

Before tool results are written to task history, the server removes any pending approval request payloads and summarizes screenshot data URLs into redacted metadata. This keeps history useful for audit and debugging without persisting raw screenshot images.

Task artifacts are created only from persisted terminal tool results or persisted task output. They can contain page-derived content such as extracted links, text, or sanitized snapshots, so they remain local by default and are exposed only through the loopback API and extension UI. Artifact creation and deletion audit payloads include ID, source, kind, MIME type, byte length, and title metadata, but not artifact content. Deleting an artifact removes that reusable derived output while keeping the task run and original summarized tool history intact for audit continuity.

Task deletion removes the task run, summarized tool results, persisted task output, and derived artifacts. The deletion audit event stores only task ID, prior status, plan/result/artifact counts, and revoked approval count; it does not retain the user task text or page-derived content.

Published page snapshots are sanitized again on the server before local persistence. The store keeps the latest snapshot overall and the latest snapshot per tab ID, so `GET /v1/page/snapshot` can recover after an agent restart. `DELETE /v1/page/snapshot?tabId=...` clears the saved snapshot for one tab and also clears the global latest pointer when it points to that tab; `DELETE /v1/page/snapshot` clears all saved snapshots. Snapshot update and clear audit events record only metadata such as tab ID, element/link counts, scope, and cleared record counts.

Provider settings saved from the options page are also stored in SQLite. OpenAI-compatible API keys are stored locally and redacted from `GET /health` and `GET /v1/provider-config`; operators can avoid storing API keys by using `OAB_PROVIDER` and related environment variables instead.

The new tab launch draft is stored only in extension local storage, expires quickly, and is removed when the side panel reads it. Loading a draft only pre-fills the side panel composer; page snapshots, memory context, provider calls, and task persistence still start only after the user sends the task from the side panel.

The dev browser launcher writes `open-agent-config.json` into the built extension output so a launched extension can find the selected loopback agent URL. This file contains only `agentBaseUrl`; an explicit Options-page `agentBaseUrl` stored in browser extension storage takes precedence.

Memory request and confirmation events are audited by ID and tags only; the audit payload does not duplicate the memory content.

Memory deletion removes the confirmed memory row from SQLite. The deletion audit payload includes only the memory ID and tags, not the memory content.

Approval rejection events are audited by approval ID, task/step IDs, tool call ID, tool name, and optional user-facing reason. They do not execute the rejected browser action. Rejection audit is written only after the request has been matched to the pending approval and any supplied task/step binding has been validated.

Provider-supplied approval reasons are treated as untrusted page/model output. They may be shown as a provider note, but the primary approval reason is generated locally from the validated tool call and risk classifier.

Task cancellation events are audited by task ID, optional user-facing reason, and the count of approval tokens revoked for that task plan. Canceling a task revokes outstanding task approvals, prevents later tool results from mutating that task, and does not execute any browser action.

Bulk task cancellation events are audited by canceled task IDs, optional user-facing reason, canceled task count, and revoked approval count only. They do not duplicate task messages, page text, tool results, artifacts, or provider output.

Confirmed memory can be included in chat planning and provider prompts after the user has saved it. Chat plan audit payloads record only memory counts and planning metadata, not the memory content itself.

`WS /v1/events` streams task and audit metadata to connected local clients. It does not add external telemetry, and it inherits the same payload restrictions as the audit log.

Browser requests to the local API are origin-checked. Missing `Origin` headers are allowed for local CLI/test clients, `chrome-extension://` and `moz-extension://` origins are allowed for extension UI, and ordinary `http://` or `https://` page origins are rejected unless explicitly listed in `OAB_ALLOWED_ORIGINS` for development.
